Purpose of the platform:
Users are meant to login with social media and subscribe so that new posts will be pulled to update the avatar. The social media that is connected will pull initial data to create the avatar database. Then the avatar that is created is meant to be used for email, text, tweet, discord (with audio voice bot), analysis, ui messages, twitch bots, slack. generative images of the users avatar is meant to be shared in place of an emoji when emotions are updated. reference images and audio is meant to be uploaded with streamlit and used for target identification and generative ai. messages are meant to be metered to prevent costs. there are meant to be tiers of subscriptions to allow for higher quality models such as adapter training and use. There is meant to be a training of behavioral traces to adjust the choices of the model. There is meant to be menu ordering to support the growth of the application to supplement costs for all users using social media.

Prerequisites / Infrastructure
Before the pipeline runs at all, three things must be in place. Users authenticate via social media, and their accounts are crawled for content on initial setup, then subscribed to for ongoing updates. Each user is limited to one personal avatar and must be verified with name, phone number, address, one social media account, and email. Deep research then automatically researches facts about the target, with human verification of facts and sources before they enter the pipeline.

Phase 1 — Data Ingestion (Step 1) ✅ Complete
The /update_avatar_identity_with_media endpoint accepts files and URLs from authenticated users. Files are read, their MIME type is captured, and they are packaged into media_items with user and assistant context. This feeds the process_media_graph subgraph. Anonymous user support is live via get_current_user_or_anonymous_user.

Phase 2 — Content Classification (Step 2) ✅ Complete (partial)
Two classification passes run on every text document. First, a ProprietaryContentClassification check determines whether the content has a single identifiable target or is generic/proprietary. If proprietary, it routes directly to the vectorstore. If a target exists, a TextualSituationalAwareness model classifies the situation as single_speaker, q_and_a_dialogue, multi_speaker, or other. For single-speaker content, a further MonologuePresentationOrSeriesOfQuotes sub-classification determines whether it is a flowing monologue or a series of discrete quotes/tweets. The ContentSituationClassification model adds biographical_facts and presentation as additional situation types.

Phase 3 — Named Speaker Formatting (Step 3) 🔲 Schema complete, routing incomplete
Conversational content is converted into the intermediate {speaker, content} message list format, with narrator turns for context and actions in *asterisks*. The NamedSpeakerMessageFormat class and NAMED_SPEAKER_MESSAGE_FORMAT_SYSTEM_PROMPT are built. The routing branches for q_and_a_dialogue and multi_speaker in process_text_to_document are currently stubbed as TODOs.

Phase 4 — Target Identification & Role Conversion (Step 4) 🔲 Schema complete, integration pending
The target individual is identified in the named-speaker message list, their turns are relabelled assistant, and all other speakers become user. The TargetIdentificationInText, RoleConvertedMessageFormat, and TARGET_IDENTIFICATION_AND_ROLE_CONVERSION_SYSTEM_PROMPT are built. The adapter training formatters llm_single_turn_dataset, llm_multiturn_dataset_one_conversation, and langsmith_dataset are implemented in formatting.py and consume this output.

Phase 5 — Identity Analysis (Step 5) 🔲 Schema complete, integration pending
All target messages are analyzed for a rich identity profile across 17 dimensions. Each dimension has its own independent Pydantic model and system prompt covering: name, description, identity, history, emotions, beliefs, values, opinions, goals, wants, needs, fears, problems, flaws, strengths, secrets, and relationships. The GeneralCharacteristicExtraction aggregate model runs all 17 in a single pass. The CHARACTERISTIC_EXTRACTORS registry allows the pipeline to iterate all dimensions programmatically. OCEAN analysis (perform_ocean_analysis) is already wired in for SeriesOfDistinctQuotes content. Full wiring for all other content types is pending.

Phase 6 — Synthetic Question Generation (Step 6) ✅ Complete
For monologues, presentations, and tweet series — where there is no natural conversational partner — create_question_list generates a synthetic prompt for each ground-truth target statement. This produces question-answer pairs that feed into the adapter training format alongside real dialogue.

Phase 7 — Adapter Training Data Storage (Step 7) 🔲 Partially complete
Training-formatted data (both ground-truth dialogue and synthetic QA pairs) is saved for fine-tuning a LoRA/adapter. The llm_single_turn_dataset and llm_multiturn_dataset_one_conversation formatters are ready. The training loop, adapter storage, and adapter attachment at avatar selection time are future milestones.

Phase 8 — Quality Analysis & Evaluation (LangSmith) 🔲 Planned
This phase instruments the entire pipeline and avatar runtime for continuous debugging, evaluation, testing, and improvement using LangSmith. It has four sub-tracks that run in parallel.
Tracing. Every pipeline run — classification, formatting, identity extraction, question generation, adapter training formatting, and inference — is instrumented with the @traceable decorator or REST API logging so that nested calls within tools are captured as unified traces. Custom run names are assigned at each stage for UI clarity. Trace links are surfaced in development tooling so any execution path can be inspected, annotated, or added to a dataset directly from its unique UI.
Testing & Evaluation. The langsmith_dataset formatter already outputs in the required format. On top of that, the following evaluation methods are wired in: prompt iteration regression tests comparing multiple system prompt versions against the same dataset; chatbot simulation evals where a synthetic user is given a task and the avatar is scored on how well it fulfills it without breaking character; single-turn and multi-turn conversation evals treating each dialogue turn as an independent data point; extraction chain evals measuring similarity between the 17-dimension GeneralCharacteristicExtraction outputs and reference labels using JSON evaluators; and agent trajectory evals comparing the sequence of pipeline steps taken against an expected sequence to grade correct tool and classifier usage. VADER sentiment scoring against the ground-truth baseline quote data is applied as a post-hoc metric using compute_test_metrics so it can be backfilled onto existing test results without re-running the model.
Feedback & Monitoring. User feedback signals are captured from the /message and /chat endpoints and logged to LangSmith. An automated feedback pipeline runs as a batch job against production traces to compute quality metrics continuously. A real-time async callback triggers automated feedback generation for every live run, flagging hallucinations in avatar responses against retrieved vectorstore context. Preferred and rejected responses are annotated in the LangSmith annotation queue.
Optimization. Preferred response traces are used for prompt bootstrapping — an LLM optimizer rewrites the system prompt based on accumulated feedback, with the avatar identity prompt as the primary optimization target. The persona-specific bootstrapping pattern is applied iteratively, refining the avatar system prompt to better match the ground-truth voice and identity profile of the target individual. Automated few-shot curation selects the most informative examples from the annotation queue based on performance metrics and injects them into the system prompt. Fine-tuning export pipelines convert approved LangSmith run data into OpenAI or LLaMA fine-tuning format, feeding back into Phase 7 adapter training. Lilac is used for dataset curation — detecting near-duplicates, checking for PII, and enriching datasets — before fine-tuning runs.

Phase 9 — Inference / Avatar Runtime ✅ Operational (Updated)
The /message (selected avatar) and /message/{assistant_id} (direct avatar addressing) endpoints are live. Both support optional file attachments, anonymous users, and persistent thread management via AsyncPostgresSaver checkpointer. Each conversation generates a thread_id that is returned in the response and can be passed on subsequent calls to maintain context. Thread metadata including most_recent_message and conversation_title is kept current after every turn. The /conversations endpoint returns all threads for a user-avatar pair sorted newest-first, and /conversations/{thread_id}/messages returns the full message history for any thread. A response_only_workflow graph is compiled alongside the main graph for latency-sensitive paths that do not require full tool resolution.

Phase 10 — Latency Reduction 🔲 Planned
Several strategies are planned to reduce end-to-end response latency. The response_only_workflow graph (already compiled) separates fast response generation from heavier background tasks like vectorstore writes and analysis. Additional strategies include: streaming token output to the client via SSE rather than waiting for the full response; parallelising independent pipeline stages (vectorstore retrieval, memory lookup, identity context loading) using async gather; caching frequently accessed identity profiles and vectorstore embeddings at the connection pool level; reducing cold-start time by pre-compiling graphs and warming the embedding model at lifespan startup rather than on first request; and using smaller, faster models for low-stakes classification steps (proprietary content check, situation detection) while reserving larger models for identity analysis and final response generation.

Phase 11 — Stripe Metering 🔲 Planned
The current Stripe integration handles subscription creation, management, cancellation, and status verification. The next phase extends this to usage-based metering. Every avatar message, media upload, and identity analysis invocation is metered via Stripe's usage records API so that per-seat or per-call billing can be layered on top of the base subscription. Rate limiting is enforced at the FastAPI middleware level, gating requests against the user's current subscription tier and metered usage balance before they reach graph execution. Webhook handlers for invoice.payment_failed, customer.subscription.deleted, and customer.subscription.updated events keep the app_metadata subscription status in sync in real time, so access is revoked or restored immediately without polling.

Phase 12 — Email Ambient Agent (Ignore / Respond / Notify) 🔲 Planned
An ambient email agent built with LangGraph monitors an inbox and autonomously triages every incoming message into one of three actions: ignore, respond, or escalate with a notification. The agent uses a LangGraph state machine where each email enters a classification node that determines the appropriate action based on sender, subject, content, and the avatar's known identity, relationships, and preferences. For emails classified as respond, the avatar drafts a reply in the target individual's voice using the full identity profile and sends it. For escalations, the agent pushes a notification with a summary and recommended action. Human-in-the-loop interrupts allow the user to review, edit, or override any draft before it is sent. LangSmith traces every decision and response for evaluation and prompt optimization. The /handle_email endpoint is already stubbed in the webapp and wired to the email config schema.

Phase 13 — Data Analysis Agent 🔲 Planned
A deep agent built with LangChain's create_deep_agent pattern accepts uploaded data files (CSV, spreadsheet, or structured export) and autonomously performs exploratory data analysis, generates visualizations, and delivers findings. The agent executes code in a sandboxed backend (Daytona, Modal, or Runloop) so that generated scripts, plots, and reports are isolated from the application runtime. Results are shared to a configured output channel — Slack, email, or stored as artifacts for download. For the avatar use case, this agent connects to a user's health, fitness, and financial data feeds to produce regular self-awareness reports delivered in the avatar's voice. LangSmith traces every execution for quality review.

Phase 14 — Deep Research Agent 🔲 Planned (Prerequisites use now)
A multi-agent research system built with LangGraph handles deep research tasks about the target individual and about topics the avatar is asked about. The system has three layers: a scoping agent that clarifies and decomposes the research question; a research agent that uses web search and MCP tool integrations to gather sources; and a research supervisor that coordinates multiple parallel research sub-agents, synthesises findings, and produces a structured report. This agent is the primary driver of the prerequisite step — automatically researching publicly available facts about a target individual before their avatar is created — replacing the current manual deep research step. LangSmith evaluates the quality of research outputs against reference answers.

Phase 15 — Twitch Moderation Bot 🔲 Planned
A Twitch bot that connects to a streamer's channel and acts as a real-time moderation and engagement agent in the avatar's voice. The bot monitors the chat stream, responds to viewer messages addressed to the avatar, enforces moderation rules (removing spam, hate speech, and rule violations), rewards positive engagement, and surfaces noteworthy moments to the streamer as notifications. The avatar's full identity profile — personality, opinions, communication style, relationships — is loaded into the system context so responses feel authentic to the target individual. The bot uses the response_only_workflow graph for low-latency chat responses and logs all interactions to LangSmith for quality review and prompt tuning. Moderation decisions are logged separately for audit and appeal handling.

Platform / Avatar Management ✅ Complete
Full CRUD for avatars is implemented: create_avatar, modify_avatar, share_avatar, delete_avatar, list_public_avatars, list_user_avatars, select_avatar. Document management (list_avatar_documents, delete_avatar_document) is live. Stripe subscription and verification gating are in place.
